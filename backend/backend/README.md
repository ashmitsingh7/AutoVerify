# AutoVerify API (backend)

FastAPI service layer over AutoVerify Core. This layer is a pure
orchestrator: request validation, job/zip management, and mapping Core
errors to HTTP responses. It contains **no** parsing or generation logic -
that all lives in `../lib/AutoVerify/*` (Perl) and is reached only through
`../bin/autoverify_bridge.pl`.

```
Frontend (not built yet)
     │
     ▼
FastAPI (backend/app/main.py)
     │
     ▼
Service layer (backend/app/service.py) - orchestration only
     │        subprocess -> bin/autoverify_bridge.pl (JSON in/out)
     ▼
AutoVerify Core (../lib/AutoVerify/*)
```

## Local setup

```bash
cd backend
pip install -r requirements.txt
# Perl core (JSON::PP) needs no extra install - it ships with any modern Perl.
uvicorn app.main:app --reload --port 8000
```

Then visit `http://localhost:8000/docs` for interactive OpenAPI docs (auto-generated by FastAPI from `app/models.py`).

## Endpoints

| Method | Path        | Purpose                                                        |
|--------|-------------|-----------------------------------------------------------------|
| GET    | `/`         | Service info                                                    |
| GET    | `/health`   | Liveness check                                                   |
| POST   | `/validate` | Parse + run semantic validation on RTL text; returns diagnostics, no files written |
| POST   | `/analyze`  | Same underlying check as `/validate` - returns module/port/param structure for inspection tooling |
| POST   | `/generate` | Full pipeline: parse, validate, generate, zip-package. Returns a `job_id` |
| POST   | `/download` | Given a `job_id` from `/generate`, streams the module's `.zip` |

Request/response shapes are defined in `app/models.py` and enforced by Pydantic - see `/docs` for the live schema, or `/openapi.json` for the raw spec.

### Example

```bash
curl -s -X POST http://localhost:8000/generate \
  -H 'Content-Type: application/json' \
  -d '{"rtl": "'"$(cat ../examples/counter.v | sed 's/"/\\"/g' | tr '\n' ' ')"'", "options": {"num_txns": 50}}'
```

## Error handling

Core exceptions (`AutoVerify::ParserError`, `AutoVerify::ValidationError`) map to **422** - the RTL text was the problem, not the service. Everything else unexpected (bridge process failure, disk errors) maps to **500**. Unknown `job_id` on `/download` is **404**. A blank `rtl` field is **400**. Malformed request bodies (wrong types, missing required fields) are **422** via Pydantic automatically.

## Jobs

Every `/generate` call gets a fresh UUID job directory under `backend/app/jobs/<uuid>/`, containing the input RTL, generated `.sv` files, and the packaged zip. Jobs older than `JOB_TTL_SECONDS` (default 1 hour, see `service.py`) are swept opportunistically on the next `/generate` call - there's no background scheduler, to keep the service dependency-free. A job directory is never reused or overwritten (`_job_dir` raises if the UUID somehow already exists).

## Testing

```bash
cd backend
python3 -m pytest tests/ -v
```

- `tests/test_api.py` - every route, including error-status mapping
- `tests/test_service.py` - orchestration logic directly, bypassing HTTP
- `tests/test_integration.py` - concurrent job creation (thread pool, checks for id collisions / cross-job state bleed) and a 200-port synthetic module as a stand-in for a large real DUT

**Honest scope note on "concurrent request tests":** these use an in-process thread pool against `TestClient`, which proves job isolation and no shared-state corruption under concurrent orchestration - it does not exercise a real bound socket under network load (no actual concurrent HTTP connections, no ASGI-server-level contention). Load-testing the actual running server (e.g. with `hey` or `wrk` against `uvicorn ... --workers N`) would be the next step before treating this as verified for production traffic.

## Known limitations

- The Perl bridge subprocess has a 30s timeout (`service._run_bridge`); a pathological RTL file that makes the parser loop (shouldn't happen given the parser has no unbounded loops, but not formally proven) would time out as a 500 rather than hang the request indefinitely.
- No auth/rate-limiting - add before exposing this publicly.
- No persistent job store - job records live only as long as the `backend/app/jobs/` directory (i.e., the container's filesystem). A redeploy or container restart loses in-flight job history (acceptable for a stateless generation service where the zip is meant to be downloaded promptly).
